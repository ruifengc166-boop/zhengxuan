# AI 视频全流程与团队审批机制（后续版本规划）

日期：2026 年 6 月 18 日

> 说明：当前高保真原型采用“个人创作 + 用户自审”模式，不启用本文所述团队审批能力。本文仅作为下一版本的组织协作与多级审批规划。

## 一、结论

现有“资料—脚本—分镜—生成—剪辑—审核—发布”的线性流程，能够覆盖标准宣传片，但不能覆盖全部 AI 视频创作方式。

平台应升级为双层结构：

1. **模块化智能创作前台**：面向普通宣传员。选择模板后，按内容、主体、多参考、视频、声音和后期模块操作，不显示节点和连线。
2. **节点式编排引擎**：作为后台技术底座和可选专家能力。文本、图像、视频、音频、3D、后期、审查和审批均可编排、并行和复用。

模板和模块不是另一套系统，而是对后台节点图的简化呈现。普通用户做出的模块选择会由系统转换为节点、端口和模型参数。

### 前台易用性原则

- 项目中心是唯一的创作入口，项目卡片直接告诉用户“下一步做什么”；
- 用户无需理解资料、脚本、分镜和模型属于哪个系统页面；
- 进入项目后按内容、画面、视频、声音、优化、审核六步推进；
- 每一步只提出完成当前任务所必需的问题；
- 画面和视频生成阶段允许两个必要动作：“再生成一轮”和“锁定所选版本”；
- 必须真实呈现 AI 生成中、失败、审核拦截、可用候选和历史版本，不能假定一次生成成功；
- 每轮生成应展示预计费用、实际费用、模型、API、任务 ID 和成功失败数量；
- 默认只显示业务语言，例如“添加动作参考”，不显示端口、控制图或模型内部参数；
- 根据当前模型动态显示它支持的生成方式；
- 多参考素材按人物、场景、动作、运镜、风格、音频分类上传；
- 每个镜头从下拉选项中选择文生视频、首尾帧、多参考、续写、多镜头或视频编辑；
- 高级参数折叠，系统给出推荐值；
- 任务提交前显示模型、API 来源和预计费用。

## 二、参考 AniShort 后应吸收的产品结构

从 AniShort 当前公开教程可见，其核心不是简单的页面串联，而是：

- 创作画布、剪辑预览、成片审阅三个工作空间；
- 剧本节点、剧本优化、主体提取、主体生成、分镜生成、图像生成和视频生成节点；
- 角色、场景、道具从剧本中提取后成为可重复使用的主体资产；
- 节点之间通过不同类型端口传递文本、主体、图像、视频等数据；
- 同一上游素材可以连接多个生成分支，进行模型、风格和参数对比；
- 角色生成组、道具生成组等节点组可以批量处理同类资产；
- 项目按画布、集数和节点树进行层级管理；
- 画布生成结果进入剪辑预览，最终进入成片审阅。

本平台需要在这套“创作可控性”之上，增加数据等级、授权、模型白名单、内容审查和审批责任链。

## 二点五、正确的 Prompt 驱动数据流

创作流程不能设计成“选择模型—直接挑成品”。根据 AniShort 官方最新工作流，应保留以下中间产物：

```text
原始资料
→ 剧本 Prompt / 自然语言修改意见
→ 优化后的剧本
→ 分镜拆解 Prompt
→ 结构化分镜脚本
→ 主体提取与主体设定 Prompt
→ 角色、场景、道具设定图
→ 单镜头图像 Prompt
→ 文生图候选与锁定分镜
→ 单镜头视频 Prompt
→ 图生视频/多参考视频候选
→ 选片、重试、锁定
→ 剪辑与后期
```

### 图像 Prompt

每个镜头应显示：

- 分镜脚本和镜头目标；
- 自动引用的角色、场景、道具和风格主体；
- 可编辑的图像 Prompt；
- AI 优化、根据分镜重写和版本历史；
- 景别、构图、人物位置、动作、光线、风格和画幅；
- 模型、API、比例、生成数量和高级参数；
- 当前 Prompt 版本对应的候选图、失败原因和费用。

### 视频 Prompt

图片锁定后才进入视频 Prompt。视频 Prompt 不应只是复述静态画面，而应包括：

- 按时间段描述人物动作；
- 运镜方向、速度、机位和焦段；
- 镜头节奏和是否允许切镜；
- 主体、服装、场景等一致性要求；
- 首帧、尾帧、人物、场景、动作、运镜、风格和音频参考；
- 声画同出、旁白、对白或口型要求；
- 常见失败项的负向约束；
- 模型、生成模式、时长和候选数量。

每次抽卡必须绑定 Prompt 版本、参考素材快照、模型参数、供应商任务 ID、费用和审核结果。

## 四、节点数据模型

## 五、审批是否要覆盖每一个环节

答案是：**不应默认每一个环节都审批。**

每次提示词修改、抽卡或重新生成都要求审批，会严重降低生产效率，并导致审批人机械点击。

应采用三种闸门：

### 5.1 必审闸门

原则上包括：

1. 项目立项和数据等级确认；
2. 对外正式口径或脚本定稿；
3. 高风险权利事项确认，如真人肖像、克隆声音；
4. 专业内容审核，如医疗、应急、教育；
5. 正式成片终审和发布授权。

不同单位可关闭不需要的立项或脚本审批，但正式发布闸门不能被普通用户关闭。

### 5.2 条件触发审批

出现以下情况时动态增加：

- L2 及以上数据；
- 领导干部、公众人物或特定真人；
- 未成年人、患者、学生或敏感个人信息；
- 地图、国旗国徽、党旗党徽和重要机构标志；
- 医疗建议、突发事件、民族宗教等高风险主题；
- 声音复刻、换脸、动作迁移等能力；
- 使用用户自带 API 或改变数据出境、存储边界；
- 使用未完成授权的资产；
- 机审发现 R2、R3 或 R4 风险；
- 已批准内容发生实质性修改。

### 5.3 可选确认节点

用于创作团队内部：

- 美术设定确认；
- 角色定妆；
- 分镜确认；
- 样片确认；
- 配音确认；
- 客户或业务部门意见。

它们是“确认”而不是法定或正式审批，可以并行、批注或跳过。

## 六、审批对象不是活节点，而是版本快照

发起审批时，系统冻结一个 Approval Snapshot，包括：

- 被审批的脚本、资产、镜头或成片；
- 所有关联的来源和授权；
- 节点图版本；
- 模型、参数和任务 ID；
- 自动审查结果；
- 与上一个批准版本的差异；
- 发起人和审批理由。

审批期间创作者可以继续在新分支工作，但不能偷偷改变正在审批的版本。

修改分为：

- 非实质性修改：错别字、时间线小调整等，可由策略决定是否重审；
- 实质性修改：政策口径、人物、事实数据、声音、主要画面或模型数据边界变化，必须重新审批受影响的节点和下游结果。

## 七、审批人员从哪里进入

### 7.1 统一审批工作台

侧边栏提供“审批工作台”，审批人登录后默认看到：

- 我的待办；
- 即将超时；
- 高风险任务；
- 我已处理；
- 我发起的；
- 抄送我的；
- 按单位、部门、项目和审批类型筛选。

### 7.2 项目内审批入口

项目顶部显示当前闸门。制作人可点击“发起审批”，审批人通过深链接直接定位到：

- 具体脚本段落；
- 具体资产和授权；
- 具体分镜或时间点；
- 具体风险；
- 完整成片。

### 7.3 消息、移动端和 OA

- 站内消息和待办角标；
- 短信或企业消息提醒；
- 移动端轻审批；
- OA 待办、统一身份认证和电子签章集成；
- 邮件只发送通知，不在邮件正文暴露敏感内容。

## 八、审批人界面必须提供什么

审批人不应进入复杂的创作画布寻找问题，而应看到一个“审批阅读模式”：

1. 本次请求批准什么；
2. 与上一版改了什么；
3. 关键内容和成片预览；
4. 自动审查发现了什么；
5. 事实来源和授权是否完整；
6. 哪些内容需要本人承担专业责任；
7. 上游和下游影响范围；
8. 通过、附条件通过、退回指定节点、转交、会签、加签和拒绝。

退回必须指定：

- 退回节点；
- 问题位置；
- 修改要求；
- 是否需要重新经过前置审批；
- 截止时间和责任人。

## 九、推荐的默认审批策略

### 普通公开宣传项目

项目声明确认 → 脚本定稿 → 成片终审 → 发布授权。

### 医疗科普项目

项目声明确认 → 脚本事实审查 → 医务处专业审核 → 成片宣传审核 → 发布授权。

### 含真人和声音项目

项目声明确认 → 肖像/声音授权确认 → 脚本审核 → 样片确认 → 成片终审。

### 高风险政务项目

项目立项 → 数据/保密审查 → 脚本口径审核 → 视觉规范审核 → 专业部门会签 → 领导终审 → 发布授权。
